# Crypto & Indices Hourly ML Trading System

Automated machine learning trading system for crypto (BTC, ETH, SOL, XRP, DOGE) and European indices (SMI, DAX, CAC40). Generates hourly BUY/SELL/HOLD signals using ensemble ML models with walk-forward validation.

**Current best config (BTC 4h):** RF+LR+LGBM | w=100h | Set A (18 features) | 72.5% accuracy
**Horizons:** 1h (short-term) and 4h (medium-term) — separate models, features, and windows per horizon

### Key Rules for Editing This Codebase

1. **Never use Mode A for 1h prediction.** Set A/B features are 4h-optimized. Always use Mode D for 1h
2. **CSV merge logic is critical.** When saving to crypto_hourly_best_models.csv, always filter by BOTH coin AND horizon. Wrong filter = overwrite the other horizon's config
3. **feature_set check must include E2/E3.** Use `fs in ('D', 'E2', 'E3')` everywhere. If you only check `fs == 'D'`, Mode E results fall back to hardcoded Set A features
4. **generate_signals() needs feature_override.** When feature_set is D/E2/E3, the caller must pass `feature_override=config['optimal_features'].split(',')`. Without this, it defaults to Set A
5. **Live trader imports from crypto_trading_system.py.** The filename must be exact (Windows is case-insensitive but Python imports are case-sensitive on Linux)
6. **Labels are relative to rolling median, not absolute.** `label = 1` means "return in next H hours is above the 168h rolling median return", not just "price goes up"
7. **No model persistence.** The system retrains from scratch every prediction. No .pkl files. This is intentional — ensures fresh models on latest data
8. **Each horizon = completely independent model.** Different features, model combo, window, labels. Never mix them
9. **run_loop() calls run_once()** — never duplicate signal logic in run_loop
10. **All timestamps in live trader use Europe/Zurich** via `_to_local()` helper

### Architecture in 30 Seconds

```
Data (Binance/yfinance) → ~124 features (technical + macro + sentiment)
  → Walk-forward ML (RF/GB/LR/LGBM ensembles, retrained every hour)
  → BUY/SELL/HOLD signal + confidence %
  → Telegram notification + backtest chart
```

Each (asset, horizon) has its own: feature subset, model combo, training window, labels. All stored in one CSV row in `crypto_hourly_best_models.csv`.

## Key CSV: crypto_hourly_best_models.csv

This is the central configuration file. Each row = one (asset, horizon) combination:

```csv
coin,best_window,best_combo,accuracy,models,...,feature_set,horizon,optimal_features
BTC,100,RF+LR+LGBM,72.5,RF+LR+LGBM,...,A,4,
BTC,200,GB+LGBM,73.1,GB+LGBM,...,D,1,"logret_1h,rsi_14h,price_velocity_1h,..."
```

Columns that matter:
- `feature_set`: A, B, D, E2, or E3 — determines which features to use
- `horizon`: 1 or 4 — determines which labels (1h or 4h ahead) to train on
- `optimal_features`: comma-separated feature list (only for D/E2/E3; empty for A/B)
- `models` / `best_combo`: model names joined with + (e.g., "RF+LR+LGBM")
- `best_window`: training window in hours

Feature set logic:
- A or B → uses hardcoded FEATURE_SET_A (18) or FEATURE_SET_B (14) constants
- D, E2, or E3 → uses the optimal_features column (custom per asset+horizon)

The live trader reads ALL rows for the requested asset, one per horizon, and generates independent signals for each.

## Modes Quick Reference

| Mode | Purpose | Time | When to Run |
|------|---------|------|-------------|
| **A** | Tests Set A vs Set B, picks winner | ~3-6h | Monthly (4h only — Sets are 4h-optimized) |
| **B** | Quick signals from saved models | ~2 min | Daily |
| **C** | Set A vs Set B comparison only | ~3-6h | Ad hoc |
| **D** | Full pipeline: 124 features → analysis → optimal subset → diagnostic | ~3-8h | Quarterly / new horizon |
| **E** | Iterative refinement of Mode D | ~1-4h | After Mode D |

### Mode A — Full Review
Tests Set A (18 features) vs Set B (14 features), picks winner, runs 75-config diagnostic, saves best. **WARNING: Set A/B are 4h-optimized only. For 1h, ALWAYS use Mode D instead.**

### Mode B — Quick Run
Loads saved best models from CSV. Downloads latest data, generates signals for last 200 hours per horizon, runs backtest, saves charts. Fast — use daily.

### Mode C — Compare
Diagnostic comparison of Set A vs Set B. Saves winning models. Mostly superseded by Mode A.

### Mode D — Full Pipeline
Builds all ~124 features → 5 statistical tests (LGBM importance, permutation importance, ablation, reduced sets, scoring) → determines optimal subset → 75-config diagnostic with optimal features → saves best → signals + charts. **Always use for new horizons** since Set A/B are 4h-specific.

### Mode E — Iterative Refinement (requires Mode D or previous E run)
Menu asks: "2nd pass only" or "2nd + 3rd pass"

**2nd pass (saves as E2):** Leave-one-out feature pruning (finer step=24h vs 72h), add-back test (tries re-adding dropped features from full 124), expanded window grid around winner (±100h in steps of 25h).

**3rd pass (saves as E3):** Feature interaction products of top-5 feature pairs (e.g. logret_1h × rsi_14h), ultra-fine window grid (±40h, step=10h), finest evaluation step (12h).

Typical improvement: +1-3% over Mode D.

## System Architecture

### Features (~124 total)

**43 Technical features** (built in `build_hourly_features()`):
- 12 log returns: logret_1h through logret_240h
- 6 momentum spreads: spread_Xh_Yh (fast vs slow returns)
- 4 MA ratios: SMA ratios at various lookbacks
- 7 oscillators: RSI(14), Stochastic, Bollinger Band position, Z-score(50h), ATR%(14h), intraday range, volume ratio
- 3 volatility: volatility_24h/48h/168h
- 4 time cyclicals: hour_sin, hour_cos, dow_sin, dow_cos
- 7 derivatives: price_velocity_1h/4h, price_accel_1h/4h/12h/24h, price_jerk_1h

**~80 Macro features** (built in `build_all_features()`, reads from macro_data/):
- VIX: level, regime (low/med/high/extreme), changes, MA ratios
- DXY: level, changes, MA ratios
- S&P500/Nasdaq: momentum, MA ratios
- Gold: correlation with asset
- US10Y: yield changes
- EUR/USD: volatility features
- Fear & Greed Index: level, changes, MA ratio, extreme flags
- Cross-asset: relative strength vs BTC, rolling correlations, dispersion

### Labels
```python
median_return = df['logret_Xh'].rolling(168).median()
label = 1 if logret_Xh > median_return else 0
```
Where X = horizon (1 or 4). Adaptive threshold — not just "price goes up".

### Models
All from scikit-learn / LightGBM:
- RF: RandomForest (n=300, depth=4)
- GB: GradientBoosting (n=300, depth=3)
- LR: LogisticRegression (max_iter=1000)
- LGBM: LightGBM (n=300, depth=4, GPU-accelerated)

All 15 combinations tested: 4 solo + 6 pairs + 4 triples + 1 quad.

### Walk-Forward Validation
No future leakage. For each prediction: train on last `window` hours → predict next candle → step forward. In live mode, retrains from scratch every hour on latest data.

### Scoring
`combined_score = accuracy × profit_factor` — must be both accurate AND profitable after 0.09% fees.

### Signal Generation (live)
Multi-horizon. Each horizon has independent model/features/window. Live trader sends combined Telegram:
```
BTC  🟢 1h: BUY (82%) | 🔴 4h: SELL (61%)
💰 Price: $71,540.00
📈 RSI: 45.2
🤖 1h: GB+LGBM | w=200h | Set D | 73.1%
🤖 4h: RF+LR+LGBM | w=100h | Set A | 72.5%
```
